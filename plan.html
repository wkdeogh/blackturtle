Black Turtle
구현 계획

미국 주식 투자 판단에 필요한 FRED 매크로 지표와 선택한 X 계정의 기업 언급·감성을 하나의 모바일 대시보드에 저장해 보는 개인용 웹앱.

01
제품의 가장 중요한 규칙
페이지를 열거나 새로고침하는 것만으로 외부 API를 호출하지 않는다. 화면은 항상 Supabase에 저장된 “마지막 성공 스냅샷”을 읽고, 사용자가 데이터 갱신을 눌렀을 때만 새 데이터를 수집한다.

첫 버전의 범위

지금 필요한 핵심 루프를 작게 완성하고, 이후 기능은 데이터 구조를 깨지 않고 덧붙인다.

1

페이지별 현황 보기

FRED 매크로, X 분석, 갱신 히스토리, X 계정 설정을 독립 페이지로 나누고 모바일 상단 탭으로 빠르게 이동한다.

2

소스별 수동 갱신

FRED 버튼은 FRED만 호출한다. X 원문 수집과 저장 원문의 LLM 재분석을 분리해 테스트하며, 실행 중 중복 클릭은 막고 실패하면 기존 스냅샷을 유지한다.

3

처음에는 빈 화면

성공 스냅샷이 0개면 가짜 수치를 만들지 않고 “아직 갱신된 데이터가 없습니다”와 첫 갱신 버튼을 보여준다.

동작 구조

조회 경로와 갱신 경로를 분리하면 접속할 때 비용이 발생하지 않고, 실패에도 화면이 안정적이다.

Trigger사용자 버튼로그인된 사용자만 POST /api/refresh 실행
Queue지속 실행 등록Vercel Workflow에 넣고 브라우저에는 즉시 실행 ID와 상태 반환
CollectFRED 또는 X선택한 페이지의 API만 호출하고 다른 소스의 마지막 데이터는 그대로 보존
Analyze기업·감성 분석게시물별 기업을 식별하고 기업별 positive / neutral / negative 판정
Publish원자적 공개전 과정 성공 후에만 새 스냅샷을 현재 버전으로 전환하고 히스토리에 보관

평상시 조회

브라우저 → Next.js 서버 → dashboard_state.published_snapshot_id → 한 개의 읽기 최적화 JSON 스냅샷. 실행 중에는 refresh_runs만 폴링하며 외부 데이터 API는 호출하지 않는다.

실패 시 원칙

선택한 FRED 또는 X 갱신이 실패하면 실행 기록만 failed로 남긴다. 기존 성공 스냅샷은 바꾸지 않고 오류와 재시도 버튼을 표시한다.

모바일 화면 구성

첫 화면에서 “언제 갱신됐는지, 시장 환경이 어떤지, 어떤 기업이 많이 언급됐는지”가 바로 보여야 한다.

빈 상태

“아직 저장된 데이터가 없습니다. 첫 갱신은 계정 수와 게시물 양에 따라 시간이 걸릴 수 있습니다.”라는 설명과 하나의 명확한 첫 데이터 갱신 버튼만 제공한다.

데이터 모델

원본·분석 결과·화면용 스냅샷을 분리해 재분석과 향후 기능 추가를 쉽게 한다.

테이블핵심 역할주요 필드 / 제약
refresh_runs갱신 실행 이력과 상태source, status, stage, workflow_run_id, started_at, finished_at, error_summary · active 실행은 전체에서 1개
dashboard_state현재 공개 스냅샷 포인터primary 단일 행, published_snapshot_id · 성공 시 트랜잭션으로 교체

권장 FRED 시작 세트

DGS2, DGS10, T10Y2Y, FEDFUNDS, CPIAUCSL, UNRATE, PAYEMS, M2SL, VIXCLS. 실제 표시 단위와 갱신 주기가 서로 다르므로 지표별 최신 관측일도 함께 노출한다.

집계 기준

한 게시물이 두 기업을 언급하면 두 기업에 각각 1회로 센다. 같은 게시물을 재수집해도 x_post_id 고유 제약으로 중복 집계하지 않는다. 전체 글이 아닌 각 기업을 향한 문맥으로 감성을 판정한다.

기업 식별과 감성 판정

완전 자동 판정으로 시작하되 결과를 추적하고 나중에 다시 분석할 수 있어야 한다.

1

기업 식별

LLM이 $NVDA 같은 cashtag, 티커, 기업명, 제품명·경영진 언급을 문맥에서 식별한다. 상장기업이 아니거나 불명확한 대상은 제외한다.

2

기업별 판정

각 후보에 대해 positive | neutral | negative, 신뢰도, 짧은 근거를 구조화된 형식으로 반환한다. 불확실하면 neutral 또는 미분류로 둔다.

3

재분석·검수

분석 모델을 저장하고, X API를 다시 호출하지 않은 채 현재 저장된 게시물 전체의 기업 감성과 주제를 다시 분석할 수 있다.

LLM 분석 엔진

OpenAI Responses API와 Structured Outputs를 사용해 게시물별 기업·감성·신뢰도·근거와 전체 한국어 번역을 한 번에 받고, 전체 게시물의 반복 주제를 관련 게시물 수 기준으로 정리한다. 기업 분류·번역은 gpt-5-nano, 주제 종합은 gpt-5-mini를 기본으로 사용한다. 테스트 단계에서는 X 원문 수집과 LLM 분석을 분리하고, 재분석은 저장된 게시물만 사용한다.

보안·안정성 원칙

배포 주소가 공개되어도 남이 데이터 갱신을 실행하거나 비밀키를 가져갈 수 없게 한다.

비밀번호 게이트

최초 접속 시 SITE_PASSWORD를 확인한다. 성공하면 서버가 90일짜리 서명 쿠키를 발급하며, 비밀번호 원문은 브라우저 저장소나 쿠키에 넣지 않는다.

키는 서버에만

FRED, X, OpenAI, Supabase secret 키와 사이트 비밀번호는 Vercel 환경 변수로만 저장한다. 브라우저에서 외부 API나 Supabase에 직접 접속하지 않는다.

RLS + 이중 서버 검증

Supabase RLS를 모든 테이블에 켜고 public 정책을 만들지 않는다. 페이지 앞단의 proxy.ts와 갱신 Route Handler가 각각 서명 쿠키를 검증한다.

비용·장애 방어

갱신 잠금, 계정별 최대 페이지 수, 전체 게시물 상한, API timeout/retry, 분석 batch 크기와 예산 상한을 둔다. 실패 로그에는 토큰·키·게시물 전문을 남기지 않는다.

구현 로드맵

각 단계마다 독립적으로 확인 가능한 결과물을 만든다. 예상 시간보다 완료 조건을 기준으로 진행한다.

MVP 완료

완료 조건

아래 조건을 통과하면 MVP의 핵심 동작이 완성된 것으로 본다.

  • 데이터가 0건이면 빈 상태와 첫 갱신 버튼만 보인다.
  • 페이지 접속·새로고침만으로 FRED/X/AI API가 호출되지 않는다.
  • 갱신 성공 후 재접속해도 동일한 마지막 스냅샷이 보인다.
  • 이전 성공 스냅샷은 히스토리에서 다시 볼 수 있고 설정한 최대 개수까지만 보관된다.
  • 갱신 도중 기존 스냅샷이 사라지지 않는다.
  • 갱신 직후 새로고침·페이지 이동·배포가 발생해도 작업이 지속되고 진행 상태를 다시 볼 수 있다.
  • API 하나가 실패하면 이전 스냅샷을 유지하고 오류를 보여준다.
  • 갱신 버튼을 빠르게 두 번 눌러도 작업은 하나만 실행된다.
  • 같은 X 게시물을 다시 받아도 언급량이 증가하지 않는다.
  • 한 글의 서로 다른 기업 감성이 독립적으로 저장된다.
  • 로그인하지 않은 사용자는 화면·갱신 API에 접근하지 못한다.
  • 비밀번호 원문은 쿠키, localStorage, 브라우저 JavaScript에 저장되지 않는다.
  • 로그인 성공 기기는 서명된 HttpOnly 쿠키로 90일 동안 기억된다.
  • 브라우저 코드와 응답·로그에 secret key가 노출되지 않는다.
  • 360px 화면에서 가로 스크롤 없이 핵심 카드와 버튼을 쓸 수 있다.
  • 각 수치에 데이터 기준 시각 또는 관측일이 표시된다.
